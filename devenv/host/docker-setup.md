# Docker Setup

I'd like to try to install docker from release binaries instead of using package managers, this is motivated by

- the package configuration for the docker engine (docker-ce) starts the service immediately, this mean
  docker engine will start at least one with the default configuration, which may leave some unexpected
  system level effects, e.g. the iptables rules, this may be addressed by placing a daemon configuration
  file in place in advance, but not sure whether that really works
- apt upgrade may stop docker service and causing all containers to crash, not really have seen this but
  should be avoided

official document provides https://docs.docker.com/engine/install/binaries/, be careful that the
beginning "important" note is also incorrect and full of security hallucinations, the package manager
installed softwares also don't have automatic security updates if you don't manually run apt upgrade or
configure automatic apt upgrade, does anyone really configure automatic apt upgrade? that sounds really
extremely dangerous to system stability, it is true that the static binaries are statically linked, but
the non-static linked binaries are *also* nearly static, which only dynamically link libc, so system
level package updates nearly don't provide security update to package manager installed docker, this is
yet another addition to docker and its official document's countless error about security considerations,
by the way, this section also recommends docker desktop, which is also incorrect as I described earlier,
like the network functionality is completely different from normal linux installation approaches

UPDATE: there is automatic upgrade, see https://wiki.debian.org/PeriodicUpdates
and https://ubuntu.com/server/docs/how-to/software/automatic-updates/, does anyone really use these?

to avoid the not sure statically build binaries have less feature than package manager binaries, you
can use package manager binaries from package binaries from https://download.docker.com/linux/,
like for current stable debian https://download.docker.com/linux/debian/dists/trixie/pool/stable/amd64/

so the docker-setup.py is implemented to install and upgrade docker binaries, other considerations:

- currently these deb files total size is 85m, they are already compressed so recompress does not reduce size, all
  binary files are huge so cannot merge them together to eliminate dependency files, currently the cloud server is
  able to access the deb files source url so the script is designed to run there and download files there, if in
  future the source url is not available you need to make a make script to bundle related files
- only include containerd.io, docker-ce, docker-ce-cli and docker-compose-plugin, exclude rootless mode to avoid
  security vulnerabilities in rootless mode, exclude buildx plugin because currently not enough performance to run
  actual building, skip this to reduce download and processing time
- exclude all documents, man pages, legecy service configurations and shell completions, which make the install file
  list simple and clear
- include service configuration files for convenience, note that although docker group is not used in other parts of
  the docker programs, the docker socket unit explicitly declared the group so you need to remove the directive if
  you are not creating and using the docker group
- need to manually enable ip forwarding if firewall backend is nftables, in /etc/sysctl.d/90-ip-forwarding.conf:
  net.ipv4.ip_forward = 1
  net.ipv6.conf.all.forwarding = 1
  net.ipv6.conf.default.forwarding = 1
  reload with sysctl -p /etc/sysctl.d/90-ip-forwarding.conf, check with sysctl net.ipv4.ip_forward
- by the way, docker-in-docker images https://hub.docker.com/_/docker include a cli image which is
  https://github.com/docker-library/docker/blob/master/29/cli/Dockerfile (this is auto generated dockerfile, not
  actual source code of this image, lazy to find that) (the 29 in path is likely to invalidate after version update,
  change that if you come here later) indicates buildx and compose plugin can download there binaries from github
  release page https://github.com/docker/buildx/releases and https://github.com/docker/compose/releases, not used
  for now, but leave here for your information
- by the way, as the packages does not take a /etc/docker/daemon.json config file, in theory you can create the file
  in advance and assume it will take effect after docker service start, but this still not defeat the other motivation
- by the way, apt download can download the deb file, but that still need configure sources.list.d for docker
- by the way, there is no way to skip post install script defined in a package, set some environment variable
  to noninteractive only skips some of the interactive part
- by the way, windows + wsl + docker desktop also don't works if you map the socket file into an image with docker cli
  command, yet another reason to avoid docker desktop TODO confirm this after migrate to non docker desktop
